# コミュニケーションガイドライン

## 応答ルール

### 情報提供リクエストの判定

以下のキーワードを含むリクエストは情報提供のみを行い、ファイル変更を行わない：
- 「調べて」「確認して」「教えて」「知りたい」「方法があるか」

対応フロー：
1. 情報を調査・収集
2. 結果を整理して提示
3. 明示的な変更指示があるまでファイル変更しない

### 提案リクエストの判定

提案を求められた場合は、提案内容のみを提示し、実際の変更は行わない。

### 誤操作時の対応

誤って変更した場合：
1. 変更を元に戻す
2. 正しい操作を実行

謝罪のみで終わらせない。

## ドキュメント作成ルール

### 初見理解可能な文書

対象：要件定義書、設計書、実装計画書、README、技術仕様書

チャットのコンテキストを前提としない。初見の読者が理解できるように記述する。

```markdown
# ❌ コンテキスト依存
コア部分から順次拡張していく方式で、依存関係の問題を回避します。

# ✅ 初見理解可能
## 段階的実装アプローチ
基盤から順次構築する方式を採用：
1. 基盤リソース（S3、Secrets Manager）- 依存なし
2. IAM ロール - 基盤リソースの ARN を参照
3. Lambda 関数 - IAM ロールの ARN を参照

この順序で「未存在リソース参照」問題を回避する。
```

確認項目：
- 専門用語に説明があるか
- 理由が説明されているか
- 具体例があるか
- 前提知識を仮定していないか

### 段階的実装の記載

対象：実装計画書、タスク定義、デプロイ手順書

タスク記載形式：
```markdown
### Phase X: [段階名]

- [ ] X.1 [コンポーネント]の実装
  - 実装内容
  - 参照リソース
  - 検証（CFN Lint 等）
  - **人間チェック**: [確認内容]
  - _要件: [関連要件]_

- [ ] X.2 デプロイとテスト
  - デプロイ対象
  - テスト項目
  - 結果確認方法
```

確認項目：
- 各段階が独立してテスト可能か
- 段階間の依存関係が明確か
- ロールバック手順があるか

## 仕様変更時のクリーンアップ

仕様変更時は不要な記載を削除する。

対象：
- 削除された機能の要件
- 使用しなくなったコンポーネント
- 実装しないタスク
- 古いアーキテクチャ図

確認項目：
- 文書全体で矛盾がないか
- 削除要素への参照が残っていないか
- 図表が最新仕様を反映しているか

## README 作成ルール

### 前提条件セクション

プロジェクト固有の要件のみ記載する。一般的な要件（CLI、基本権限）は除外。

```markdown
## 前提条件

デプロイの前に以下を準備してください。

- [プロジェクト固有の要件 1]
- [プロジェクト固有の要件 2]
```

### トラブルシューティングセクション

プロジェクト固有の問題と解決策のみ記載。ドキュメント末尾に配置。

```markdown
## トラブルシューティング

### [問題カテゴリ]

[具体的な問題] が発生した場合：

1. 解決手順 1
2. 解決手順 2
```

## 日本語ドキュメントルール

### スペーシング

日本語と英語単語・略語の間にスペースを入れる：
- ❌ `CloudFrontディストリビューション`
- ✅ `CloudFront ディストリビューション`
- ❌ `VPCとサブネット`
- ✅ `VPC とサブネット`

### 用語の一貫性

サービス名は英語のまま保持：
- `Amazon S3 バケット`
- `AWS Lambda 関数`

技術用語は一貫した翻訳を使用：
- deployment → デプロイ
- template → テンプレート
- parameter → パラメータ

## 例外

### 内部向け技術文書
チーム共通理解のある専門用語は説明省略可。新メンバー考慮で最小限の説明は含める。

### 緊急対応文書
段階的実装より迅速性を優先する場合がある。可能な限り検証ステップは含める。
