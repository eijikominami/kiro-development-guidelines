# コミュニケーションガイドライン

## 応答スタイル

ユーザーの入力スタイルに関わらず、常に丁寧語で応答する。

## リクエスト種別の判定と対応

| キーワード | 種別 | 対応 |
|-----------|------|------|
| 調べて、確認して、教えて、知りたい、方法があるか | 情報提供 | 調査結果を提示。ファイル変更しない |
| 提案して、案を出して | 提案 | 提案内容のみ提示。実際の変更しない |
| 上記以外の変更指示 | 実行 | 変更を実行 |

誤操作時: 変更を元に戻してから正しい操作を実行（謝罪のみで終わらせない）。

## ドキュメント作成ルール

### 初見理解可能な文書

対象: 要件定義書、設計書、実装計画書、README、技術仕様書

原則: チャットのコンテキストを前提としない。

```markdown
# ❌ コンテキスト依存
コア部分から順次拡張していく方式で、依存関係の問題を回避します。

# ✅ 初見理解可能
## 段階的実装アプローチ
基盤から順次構築する方式を採用：
1. 基盤リソース（S3、Secrets Manager）- 依存なし
2. IAM ロール - 基盤リソースの ARN を参照
3. Lambda 関数 - IAM ロールの ARN を参照

この順序で「未存在リソース参照」問題を回避する。
```

確認項目:
- 専門用語に説明があるか
- 理由が説明されているか
- 具体例があるか

### 段階的実装の記載形式

対象: 実装計画書、タスク定義、デプロイ手順書

```markdown
### Phase X: [段階名]

- [ ] X.1 [コンポーネント]の実装
  - 実装内容
  - 参照リソース
  - 検証（CFN Lint 等）
  - 人間チェック: [確認内容]
  - _要件: [関連要件]_
```

確認項目:
- 各段階が独立してテスト可能か
- 段階間の依存関係が明確か
- ロールバック手順があるか

## 仕様変更時のクリーンアップ

仕様変更時は以下を削除:
- 削除された機能の要件
- 使用しなくなったコンポーネント
- 実装しないタスク
- 古いアーキテクチャ図

確認: 文書全体で矛盾がないか、削除要素への参照が残っていないか。

## README 作成ルール

| セクション | 記載内容 |
|-----------|---------|
| 前提条件 | プロジェクト固有の要件のみ（一般的な CLI、基本権限は除外） |
| トラブルシューティング | プロジェクト固有の問題と解決策のみ（末尾に配置） |

## 日本語ドキュメントルール

### スペーシング

日本語と英語単語・略語の間にスペースを入れる:
- ❌ `CloudFrontディストリビューション` → ✅ `CloudFront ディストリビューション`
- ❌ `VPCとサブネット` → ✅ `VPC とサブネット`

### 用語の一貫性

| 種別 | ルール | 例 |
|------|--------|-----|
| サービス名 | 英語のまま保持 | Amazon S3 バケット、AWS Lambda 関数 |
| 技術用語 | 一貫した翻訳 | deployment → デプロイ、template → テンプレート |

## 例外

- 内部向け技術文書: チーム共通理解のある専門用語は説明省略可
- 緊急対応文書: 段階的実装より迅速性を優先（可能な限り検証ステップは含める）
