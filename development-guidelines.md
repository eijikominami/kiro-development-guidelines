# 開発ガイドライン

## 言語規則

- ドキュメント・チャット: 日本語
- コードコメント: 英語可
- 変数名・関数名: 英語

## Git ワークフロー

### ブランチ戦略

main への直接コミット禁止。

| ブランチ種別 | 命名規則 |
|-------------|---------|
| 新機能 | `feature/<機能名>` |
| バグ修正 | `fix/<修正内容>` |
| 緊急修正 | `hotfix/<バージョン>-<修正内容>` |

### バージョニング

セマンティックバージョニング（MAJOR.MINOR.PATCH）を使用。

## 要件定義時のチェックリスト

### ユーザーフロー変更時

要件変更でユーザーの操作フローが変わる場合、以下を確認する：

| チェック項目 | 例 |
|-------------|-----|
| 各ステップ間で状態が変わるシナリオ | scan → 手動操作 → convert の間にファイル状態が変化 |
| キャッシュされたデータの有効性 | scan 時の `is_local` が convert 時も有効か |
| ユーザーの手動介入が必要な箇所 | 「手動ダウンロード」導入時、その後の処理への影響 |

### 確認手順

1. 変更前後のユーザーフローを図示
2. 各ステップで保存/参照されるデータを列挙
3. ステップ間でデータの前提条件が変わる箇所を特定
4. 変わる場合、実行時の再チェックが必要か検討
5. 必要なら要件として明示（例: 「実行時に状態を再チェックする」）

## タスク実行ルール

### 実行ファイル・CLI の実行前

ソースコードを変更した後、実行ファイルや CLI コマンドを実行する前に再インストールする。

```bash
# Python プロジェクトの場合
python3.11 -m pip install -e .

# Node.js プロジェクトの場合
npm install
```

古いバイナリを実行すると、変更が反映されず問題の切り分けが困難になる。

### 実装開始前

1. タスクに「〜に従って」の記載がある場合 → 参照ファイルを先に読む
2. 参照ファイルの要件をチェックリスト化
3. 出力フォーマット（ファイル名、データ形式等）が要件に定義されている場合 → 要件通りに実装する（「手近な属性」を使わない）

### 実装完了前

1. タスクの全指示項目を確認
2. 検証項目（CFN Lint 等）を実行
3. ガイドライン準拠を確認

### CloudFormation/SAM 作成時

1. `aws-cloudformation-guidelines.md` を参照
2. MCP で AWS ドキュメントを確認
3. CFN Lint で検証（exit code 0 必須）

## 設計時の情報検索

設計・実装前に MCP や Web 検索で確かな情報を確認する。事前学習の知識に頼らない。

### 必須確認項目

| 項目 | 確認方法 |
|------|---------|
| API/SDK の仕様 | 公式ドキュメントを検索・確認 |
| URL スキーム | 実際に動作するか検証 |
| ライブラリの機能 | 公式ドキュメントまたはソースコードで確認 |
| OS 固有の機能 | Apple/Microsoft 等の公式ドキュメントで確認 |

### 禁止事項

- 未検証の URL スキームや API を設計に含めない
- 「たぶん動く」という推測で実装しない
- 検証できない機能を要件に含めない

## 外部サービス・ライブラリの制約調査

実装前に制約を調査・文書化する。

### 調査すべき制約

- 入力: フォーマット、コーデック、ファイルサイズ上限
- 出力: 生成可能フォーマット、解像度制限
- 処理: 最大処理時間、同時実行数、レート制限
- リソース: メモリ、ストレージ、ネットワーク帯域
- コスト: 料金体系、無料枠、課金単位

### 調査手順

1. 公式ドキュメントで「Quotas」「Limits」「Supported formats」を検索
2. AWS の場合は MCP で最新情報を確認
3. requirements.md に制約を追記
4. design.md に対応方針を記載

## データ変換・マッピング設計

データ変換、フォーマット変換、システム間連携を設計する際は、マッピングテーブルを作成して全フィールドの対応関係を明示する。

### 設計手順

1. 変換元スキーマの完全列挙: 公式ドキュメント・API リファレンスから全フィールドを抽出（「使いそうなもの」ではなく「存在するもの」を起点にする）
2. 変換先スキーマの完全列挙: 出力先が期待する全フィールドを確認
3. マッピングテーブル作成: 各フィールドの対応関係を明示し、対応がないフィールドを可視化
4. フォールバック定義: 各フィールドについて「データがない場合」の代替ソースを決定

### マッピングテーブル例

| 変換先フィールド | 変換元（優先順位 1） | 変換元（優先順位 2） | 欠損時の扱い |
|----------------|--------------------|--------------------|-------------|
| capture_date | FFprobe creation_time | osxphotos capture_date | エラー |
| location | FFprobe location | osxphotos location | null 許容 |
| albums | - | osxphotos albums | 空配列 |

### 確認ポイント

| チェック項目 | 説明 |
|-------------|------|
| 全フィールド網羅 | 変換元・変換先の全フィールドがテーブルに含まれているか |
| 欠損パターン | 古いフォーマット、異なるソース等でデータが欠損するケースを想定したか |
| データ経路の影響 | クラウド同期、キャッシュ等でデータが変化するケースを考慮したか |

## 手動作業の自動化

手動で実行した作業は、再現性のあるスクリプトにする。

### スクリプト化すべき作業

- AWS リソースの作成（Lambda Layer、S3 バケット等）
- 環境セットアップ
- デプロイ手順
- データ移行

### スクリプト作成の必須要素

```bash
#!/bin/bash
set -e  # エラー時に停止

# ヘルプ表示
show_help() { ... }

# 引数解析（--help, --dry-run 必須）
while [[ $# -gt 0 ]]; do
    case $1 in
        --dry-run) DRY_RUN=true; shift ;;
        --help) show_help; exit 0 ;;
        *) echo "エラー: 不明なオプション: $1"; exit 1 ;;
    esac
done

# 一時ファイルのクリーンアップ
WORK_DIR=$(mktemp -d)
trap "rm -rf $WORK_DIR" EXIT

# 進捗表示
echo "[1/N] ステップ 1..."
```

## ドキュメント更新ルール

コード変更完了直後に README.md を更新する。「後でまとめて」は禁止。

### 更新順序

1. コード実装
2. README.md（即時更新）
3. 仕様書（requirements.md → design.md → tasks.md）
4. CHANGELOG.md

### README.md 更新トリガー

以下の変更時は必ず更新:

- 機能追加・削除
- UI/メニュー変更
- コマンドオプション変更
- 設定ファイル形式変更
- 依存関係変更
- デフォルト動作変更

## GitHub 操作

GitHub MCP サーバーを使用する。`gh` コマンドは使用禁止。

### プルリクエスト作成

```bash
# 1. コミットとプッシュ
git add -A
git commit -m "Descriptive commit message"
git push origin feature/branch-name
```

```python
# 2. MCP でプルリクエスト作成
mcp_github_create_pull_request(
    owner="owner-name",
    repo="repo-name",
    title="PR title",
    head="feature/branch-name",
    base="main",
    body="## 変更内容\n- 変更点"
)
```
