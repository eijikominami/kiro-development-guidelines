# ステアリングファイル作成ガイドライン

ステアリングファイルを作成・更新する際の原則を定義する。

## 適用範囲

このガイドラインはステアリングファイル自体の文章に適用する。成果物（README、コード、設定ファイル等）には適用しない。

## 記述原則

### 事実を淡々と述べる

| 避ける表現 | 使用する表現 |
|-----------|-------------|
| 「必須」「絶対」「必ず」「重要」 | 「〜する」「〜してはならない」 |
| 「推奨」「ベストプラクティス」 | 「〜の場合、〜する」 |
| 「警告」「危険」「注意」 | 「〜すると、〜になる」 |
| セクションタイトルの括弧書き | 括弧なしのタイトル |

### 条件を明確にする

```markdown
# 条件付きルール
デプロイボタンがある場合、Prerequisites セクションを含める

# 無条件ルール
コミットされていない変更を含むディレクトリを削除してはならない

# 不明な場合
不明な場合は、ユーザーに確認する
```

### 冗長性を排除する

削除する内容：
- 一般的な知識（Git 基本コマンド、SOLID 原則等）
- Kiro が学習済みの内容
- 業界標準で解が1つしかない内容

残す内容：
- プロジェクト固有のルール
- Kiro が間違えやすい内容
- 複数の選択肢がある内容

### 特定の事象に依存しない

過去の特定の問題や教訓は、一般化されたルールとして記載する。

```markdown
# ❌ 避ける
## 今回の教訓
1. サブモジュールのリモート設定を変更しようとした
2. `rm -rf steering` を実行してローカルの変更が失われた

# ✅ 使用する
## サブモジュール操作の原則
サブモジュールの削除は、ローカルの変更を完全に失う可能性がある
```

## ファイル管理

### 重複を避ける

- 新しいステアリングを追加する前に、既存ファイルを確認
- 関連する内容は同一ファイルにまとめる
- 他ファイルの内容を参照する場合：`ブランチ戦略の詳細は development-guidelines.md を参照`

### 分類フロー

```
新しいステアリング内容を追加する
  ↓
既存のカテゴリに当てはまるか？
  YES → 既存ファイルに統合
  NO  → 最も近いカテゴリに統合
```

### 新規内容の追加手順

1. カテゴリ判断：追加内容が属するファイルを特定
2. ファイル統合：該当ファイルに内容を追加
3. 重複排除：既存内容との重複を削除
4. 一貫性確保：ファイル全体の構成を確認
5. 個別ファイル削除：一時ファイルがあれば削除

### 既存内容の更新手順

1. 対象ファイル特定
2. 該当セクションを更新
3. 不要な記載を削除
4. ファイル全体の整合性を確認

## 文章スタイル

### 禁止表現

| 禁止 | 代替 |
|-----|------|
| 「〜が良い」「〜がベスト」 | 「〜する」 |
| 「適切に」「十分に」「なるべく」 | 具体的な基準を記載 |
| `**重要**: 必ず〜してください！！` | 「〜する」 |

### 構成順序

1. 原則：基本的な考え方
2. 手順：具体的な実行方法
3. チェックリスト：確認項目
4. 例：具体的な使用例（わかりにくい場合のみ）
5. トラブルシューティング：問題発生時の対応（該当する場合）

## チェックリスト

### 新規作成時
- [ ] 既存ファイルに統合できないか確認
- [ ] 強調表現を使用していない
- [ ] 条件を明確に記述している
- [ ] 重複する内容がない

### 更新時
- [ ] 不要な記載を削除した
- [ ] 強調表現を追加していない
- [ ] 他ファイルとの整合性を確認した
